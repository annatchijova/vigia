The contract is deliberately small.

An entry carries its position, time, event, the prior hash, and its own hash. Verification recomputes every link. There is no mystery layer: the point is a precise, inspectable integrity contract.

Entry = { index, timestamp_ms, event, prev_hash, hash } hash = SHA256(index | timestamp_ms | event | prev_hash) valid = every previous link matches AND every hash recomputes

The event order is part of the evidence. Change an entry, delete it, or move it without repairing every consequence — verification fails.

Same rules. Different trade-offs.

This is not a benchmark that compares unrelated programs. The specification and attacks stay fixed; the implementation reveals how C, Rust, and Java make memory management, concurrency, and performance visible in different ways.

The four modes below run against a ten-entry chain in all three implementations. The surprising result is not that careless tampering fails. It is that a full cascading rewrite passes — and that is the correct result for an unanchored hash chain.

Tamper-evident is not tamper-proof.

The project makes its negative result executable: after a cascading recomputation, verify() returns true. Hiding this would turn a useful integrity primitive into a misleading security claim.

The chain proves a later edit happened. It does not prove an earlier version was legitimate.

An attacker with full write access can rewrite the altered entry and every following link into a self-consistent forged chain.
